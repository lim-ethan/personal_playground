# LLM 공부 3편: RNN, LSTM, Seq2Seq

## 0. 이번 편의 목표

이번 편에서는 Transformer가 등장하기 전 NLP 모델의 핵심 흐름인 **RNN → LSTM → Seq2Seq**를 다룬다.

LLM을 깊게 이해하려면 Transformer만 보면 부족하다. Transformer가 왜 중요한지 알려면, 그 전에 쓰이던 모델들이 어떤 문제를 해결하려 했고, 어떤 한계 때문에 밀려났는지를 알아야 한다.

이번 편에서 이해해야 할 핵심 질문은 다음과 같다.

```text
RNN은 왜 등장했을까?
RNN은 문장을 어떻게 처리할까?
RNN의 한계는 무엇일까?
LSTM은 RNN의 어떤 문제를 해결했을까?
Seq2Seq는 왜 중요했을까?
왜 Seq2Seq만으로는 지금의 LLM을 만들기 어려웠을까?
```

---

## 1. Word2Vec 이후의 문제

2편에서 Word2Vec과 단어 임베딩을 봤다.

Word2Vec은 단어를 벡터로 표현할 수 있게 해줬다.

예를 들면 다음과 같다.

```text
king   → [0.21, -0.13, 0.72, ...]
queen  → [0.19, -0.10, 0.69, ...]
apple  → [-0.45, 0.33, 0.11, ...]
```

이 방식은 단어의 의미를 숫자 공간에 표현한다는 점에서 매우 중요했다.

하지만 Word2Vec에는 큰 한계가 있었다.

바로 **문장 전체의 순서와 문맥을 충분히 다루기 어렵다**는 점이다.

예를 들어 보자.

```text
나는 사과를 먹었다.
나는 사과했다.
```

여기서 `사과`라는 단어는 서로 다른 의미를 가진다.

또 다른 예시도 있다.

```text
개가 사람을 물었다.
사람이 개를 물었다.
```

두 문장은 같은 단어를 포함하지만, 단어의 순서가 바뀌면서 의미가 완전히 달라진다.

즉, 자연어를 제대로 이해하려면 단어 하나하나의 의미뿐 아니라 **단어가 어떤 순서로 등장했는지**를 봐야 한다.

이 문제를 해결하기 위해 등장한 대표적인 구조가 **RNN**이다.

---

## 2. RNN이란?

RNN은 **Recurrent Neural Network**의 약자다.

한국어로는 보통 **순환 신경망**이라고 부른다.

RNN의 핵심 아이디어는 간단하다.

```text
문장을 앞에서부터 순서대로 읽으면서,
이전까지 읽은 정보를 hidden state에 저장한다.
```

예를 들어 다음 문장이 있다고 하자.

```text
나는 오늘 점심을 먹었다
```

RNN은 이 문장을 한 번에 보는 것이 아니라 순서대로 처리한다.

```text
나는 → 오늘 → 점심을 → 먹었다
```

각 단어를 처리할 때마다 RNN은 현재 입력과 이전 상태를 함께 사용한다.

```text
현재 hidden state = f(현재 단어, 이전 hidden state)
```

조금 더 단순화하면 다음과 같다.

```text
h1 = f("나는", h0)
h2 = f("오늘", h1)
h3 = f("점심을", h2)
h4 = f("먹었다", h3)
```

여기서 `h`는 hidden state다.

hidden state는 모델이 지금까지 읽은 내용을 요약해서 들고 있는 메모리라고 보면 된다.

---

## 3. RNN이 문장을 처리하는 방식

RNN의 구조를 아주 단순하게 표현하면 다음과 같다.

```text
입력 토큰 1 → RNN Cell → hidden state 1
입력 토큰 2 → RNN Cell → hidden state 2
입력 토큰 3 → RNN Cell → hidden state 3
입력 토큰 4 → RNN Cell → hidden state 4
```

중요한 점은 RNN Cell이 매번 같은 파라미터를 사용한다는 것이다.

즉, 문장의 길이가 길어져도 같은 계산 구조를 반복해서 적용한다.

그래서 RNN은 다음과 같은 작업에 사용되었다.

```text
문장 분류
감성 분석
품사 태깅
기계 번역
음성 인식
시계열 예측
```

자연어는 순서가 중요하기 때문에 RNN은 당시 매우 자연스러운 선택이었다.

---

## 4. RNN의 장점

RNN의 장점은 분명했다.

첫 번째, **순서 정보를 다룰 수 있다.**

Word2Vec은 단어 자체의 의미를 잘 표현했지만, 문장 내 순서를 직접 다루지는 못했다. RNN은 입력을 순서대로 처리하므로 문장 구조를 어느 정도 반영할 수 있었다.

두 번째, **가변 길이 입력을 처리할 수 있다.**

문장은 길이가 일정하지 않다.

```text
안녕
오늘 날씨가 참 좋다
나는 어제 저녁에 친구와 밥을 먹고 집에 돌아왔다
```

RNN은 같은 구조를 반복해서 사용하기 때문에 길이가 다른 문장도 처리할 수 있다.

세 번째, **이전 정보를 기억할 수 있다.**

hidden state를 통해 이전 단어들의 정보를 다음 단계로 넘길 수 있다.

---

## 5. RNN의 핵심 한계: 장기 의존성 문제

RNN의 가장 큰 문제는 **긴 문장에서 앞쪽 정보를 뒤쪽까지 잘 전달하지 못한다**는 것이다.

이걸 **장기 의존성 문제**라고 한다.

예를 들어 보자.

```text
나는 어제 회사에서 긴 회의를 마치고 집에 돌아온 뒤,
잠깐 쉬다가 냉장고에 넣어두었던 사과를 먹었다.
```

여기서 `먹었다`의 대상은 앞쪽의 `사과`다.

문장이 짧으면 RNN이 이를 어느 정도 기억할 수 있다.

하지만 문장이 길어질수록 초반 정보가 hidden state를 거치며 희미해진다.

단순하게 말하면, RNN은 오래된 정보를 계속 압축해서 들고 가야 한다.

```text
앞 단어 정보 → 압축 → 압축 → 압축 → 압축 → 뒤쪽 단어
```

이 과정에서 중요한 정보가 사라질 수 있다.

---

## 6. Vanishing Gradient 문제

RNN의 장기 의존성 문제는 학습 과정에서도 발생한다.

딥러닝 모델은 학습할 때 오차를 뒤로 전파한다.

이를 **역전파**, 영어로는 **Backpropagation**이라고 한다.

RNN에서는 시간 순서에 따라 펼친 뒤 역전파를 수행한다.

이를 **BPTT**, 즉 **Backpropagation Through Time**이라고 부른다.

문제는 문장이 길어질수록 gradient가 여러 단계를 거쳐 전달된다는 점이다.

```text
오차 → t=5 → t=4 → t=3 → t=2 → t=1
```

이때 gradient가 점점 작아지면 앞쪽 단어에 대한 학습이 거의 일어나지 않는다.

이를 **Vanishing Gradient**라고 한다.

반대로 gradient가 너무 커지는 경우도 있다.

이를 **Exploding Gradient**라고 한다.

정리하면 다음과 같다.

| 문제 | 의미 |
|---|---|
| Vanishing Gradient | gradient가 너무 작아져서 앞쪽 정보 학습이 어려워짐 |
| Exploding Gradient | gradient가 너무 커져서 학습이 불안정해짐 |
| Long-term Dependency | 긴 문장에서 멀리 떨어진 단어 관계를 잘 기억하지 못함 |

---

## 7. LSTM이란?

RNN의 한계를 보완하기 위해 등장한 대표적인 구조가 **LSTM**이다.

LSTM은 **Long Short-Term Memory**의 약자다.

이름 그대로 장기 기억과 단기 기억을 더 잘 다루기 위한 구조다.

LSTM의 핵심 아이디어는 다음과 같다.

```text
중요한 정보는 오래 유지하고,
불필요한 정보는 지우고,
새로운 정보는 필요한 만큼만 반영한다.
```

RNN이 단순히 hidden state 하나로 정보를 계속 넘겼다면, LSTM은 좀 더 정교한 메모리 구조를 사용한다.

LSTM에는 크게 두 가지 상태가 있다.

```text
hidden state: 현재 출력에 가까운 정보
cell state: 장기 기억에 가까운 정보
```

특히 cell state가 중요하다.

cell state는 정보를 비교적 안정적으로 오래 전달하기 위한 통로 역할을 한다.

---

## 8. LSTM의 Gate 구조

LSTM은 정보를 조절하기 위해 **gate**라는 구조를 사용한다.

대표적으로 세 가지 gate가 있다.

```text
Forget Gate
Input Gate
Output Gate
```

각 gate의 역할은 다음과 같다.

| Gate | 역할 |
|---|---|
| Forget Gate | 이전 정보 중 무엇을 버릴지 결정 |
| Input Gate | 새로운 정보 중 무엇을 저장할지 결정 |
| Output Gate | 현재 시점에서 무엇을 출력할지 결정 |

쉽게 말하면 LSTM은 매 시점마다 이런 판단을 한다.

```text
이전 기억 중 버릴 것은 무엇인가?
새로 들어온 정보 중 저장할 것은 무엇인가?
현재 출력에 사용할 정보는 무엇인가?
```

이런 구조 덕분에 LSTM은 일반 RNN보다 긴 문맥을 더 잘 다룰 수 있었다.

---

## 9. LSTM을 직관적으로 이해하기

LSTM을 너무 수식으로만 보면 어렵다.

일단은 다음 비유로 이해하면 된다.

RNN은 작은 메모장 하나에 모든 내용을 계속 덮어쓰는 방식에 가깝다.

```text
RNN = 하나의 작은 메모장에 계속 요약해서 적기
```

반면 LSTM은 메모장을 관리하는 규칙이 있다.

```text
LSTM = 중요한 내용은 남기고, 필요 없는 내용은 지우고, 새 내용은 선별해서 적기
```

그래서 LSTM은 RNN보다 긴 문맥을 더 잘 유지한다.

하지만 LSTM도 완벽하지는 않다.

문장이 아주 길어지고 데이터가 커지면 여전히 한계가 있었다.

---

## 10. GRU란?

LSTM과 비슷한 계열로 **GRU**도 있다.

GRU는 **Gated Recurrent Unit**의 약자다.

GRU는 LSTM보다 구조가 조금 더 단순하다.

LSTM이 세 개의 gate를 사용한다면, GRU는 보통 두 개의 gate를 사용한다.

```text
Update Gate
Reset Gate
```

GRU는 LSTM보다 계산량이 적고 구현이 간단한 편이다.

다만 이번 공부의 핵심은 GRU 자체가 아니라, RNN 계열이 gate 구조를 통해 장기 기억 문제를 해결하려 했다는 점이다.

---

## 11. Seq2Seq란?

이제 Seq2Seq를 보자.

Seq2Seq는 **Sequence to Sequence**의 약자다.

말 그대로 하나의 시퀀스를 입력받아 다른 시퀀스를 출력하는 구조다.

대표적인 예시는 기계 번역이다.

```text
입력: I love you
출력: 나는 너를 사랑한다
```

입력도 문장이고, 출력도 문장이다.

즉, 문장 하나를 다른 문장으로 변환해야 한다.

Seq2Seq는 이런 문제를 풀기 위해 등장했다.

---

## 12. Encoder-Decoder 구조

Seq2Seq의 핵심은 **Encoder-Decoder 구조**다.

구조는 다음과 같다.

```text
입력 문장 → Encoder → Context Vector → Decoder → 출력 문장
```

조금 더 자세히 보면 다음과 같다.

```text
I → love → you
      ↓
   Encoder
      ↓
Context Vector
      ↓
   Decoder
      ↓
나는 → 너를 → 사랑한다
```

Encoder는 입력 문장을 읽고 하나의 벡터로 압축한다.

Decoder는 이 벡터를 기반으로 출력 문장을 생성한다.

---

## 13. Encoder의 역할

Encoder는 입력 문장을 순서대로 읽는다.

예를 들어 다음 문장을 입력한다고 하자.

```text
I love you
```

Encoder는 RNN 또는 LSTM을 사용해 단어를 순서대로 처리한다.

```text
I → love → you
```

그리고 마지막 hidden state를 문장 전체의 요약 정보로 사용한다.

이 요약 벡터를 보통 **Context Vector**라고 부른다.

```text
Context Vector = 입력 문장의 의미를 압축한 벡터
```

---

## 14. Decoder의 역할

Decoder는 Context Vector를 받아 출력 문장을 생성한다.

기계 번역 예시에서는 다음과 같이 동작한다.

```text
Context Vector → 나는 → 너를 → 사랑한다
```

Decoder는 한 번에 문장을 모두 생성하지 않는다.

이전 출력 토큰을 참고하면서 다음 토큰을 하나씩 생성한다.

```text
첫 번째 출력: 나는
두 번째 출력: 너를
세 번째 출력: 사랑한다
```

이 방식은 지금의 GPT 계열 모델이 다음 토큰을 생성하는 방식과 연결된다.

물론 구조는 다르지만, **토큰을 순차적으로 생성한다**는 관점에서는 중요한 연결점이 있다.

---

## 15. Seq2Seq가 중요했던 이유

Seq2Seq는 NLP에서 매우 중요한 전환점이었다.

그 이유는 문장을 단순히 분류하는 수준을 넘어, 문장을 다른 문장으로 변환할 수 있게 했기 때문이다.

Seq2Seq는 다음과 같은 작업에 활용되었다.

```text
기계 번역
문장 요약
질문 답변
챗봇
음성 인식 결과 생성
문법 교정
```

즉, Seq2Seq는 생성형 NLP의 중요한 기반이었다.

지금의 LLM도 넓게 보면 입력을 받아 출력을 생성한다.

그래서 Seq2Seq를 이해하면 LLM의 뿌리를 이해하는 데 도움이 된다.

---

## 16. Seq2Seq의 한계

Seq2Seq에도 큰 문제가 있었다.

가장 대표적인 문제는 **입력 문장을 하나의 Context Vector에 모두 압축해야 한다**는 점이다.

짧은 문장은 괜찮다.

```text
I love you
```

하지만 긴 문장은 문제가 된다.

```text
The security analyst reviewed multiple alerts from different endpoints,
correlated them with threat intelligence feeds,
and concluded that the suspicious process was related to a known malware campaign.
```

이 긴 문장을 하나의 벡터에 모두 압축하면 정보 손실이 생기기 쉽다.

단순하게 표현하면 이렇다.

```text
긴 입력 문장 전체 → 하나의 벡터
```

이 하나의 벡터가 병목이 된다.

이를 **정보 병목 문제**라고 볼 수 있다.

---

## 17. Attention이 필요해진 이유

Seq2Seq의 문제는 Decoder가 출력할 때 입력 문장의 모든 부분을 똑같이 압축된 Context Vector에 의존한다는 점이었다.

예를 들어 번역을 한다고 하자.

```text
입력: The cat sat on the mat
출력: 고양이가 매트 위에 앉았다
```

`고양이`를 생성할 때는 `cat`을 더 봐야 한다.

`앉았다`를 생성할 때는 `sat`을 더 봐야 한다.

`매트`를 생성할 때는 `mat`을 더 봐야 한다.

그런데 기존 Seq2Seq는 입력 전체를 하나의 Context Vector로 압축해버렸다.

그래서 이런 아이디어가 등장한다.

```text
출력 토큰을 생성할 때마다,
입력 문장의 어떤 단어를 더 중요하게 볼지 선택하면 어떨까?
```

이 아이디어가 바로 **Attention**으로 이어진다.

---

## 18. RNN/LSTM/Seq2Seq와 Transformer의 연결

Transformer는 RNN 계열의 한계를 극복하기 위해 등장했다.

핵심 차이는 다음과 같다.

| 구분 | RNN/LSTM | Transformer |
|---|---|---|
| 처리 방식 | 토큰을 순서대로 처리 | 토큰들을 병렬로 처리 가능 |
| 긴 문맥 처리 | hidden state에 의존 | attention으로 토큰 간 관계 직접 계산 |
| 학습 속도 | 상대적으로 느림 | 병렬화에 유리 |
| 핵심 구조 | recurrent cell | self-attention |
| 병목 | 순차 처리, 장기 의존성 | attention 계산 비용 |

RNN은 문장을 왼쪽에서 오른쪽으로 순서대로 읽는다.

Transformer는 문장 내 토큰들이 서로를 직접 참조할 수 있게 만든다.

이 차이가 매우 중요하다.

## 19. 지금 LLM 관점에서 다시 보기

지금 우리가 쓰는 LLM은 대부분 Transformer 기반이다.

그렇다고 RNN/LSTM/Seq2Seq 공부가 의미 없는 것은 아니다.

오히려 이 흐름을 알아야 다음 질문에 제대로 답할 수 있다.

```text
왜 Attention이 필요했는가?
왜 Transformer가 혁신적이었는가?
왜 병렬 학습이 중요했는가?
왜 긴 문맥 처리가 어려운 문제인가?
왜 Context Window가 여전히 중요한 제약인가?
```

LLM의 역사를 보면 대부분의 발전은 다음 문제를 해결하는 방향이었다.

```text
더 긴 문맥을 다루기
더 많은 데이터를 학습하기
더 빠르게 학습하기
더 정확하게 다음 토큰을 예측하기
더 좋은 출력을 생성하기
```

RNN과 LSTM은 이 흐름에서 중요한 중간 단계였다.

## 20. 보안/Agent/RAG 프로젝트와 연결하기

네가 하고 있는 MDR, RAG, Agent 시스템과도 이 개념은 연결된다.

예를 들어 보자.

### 20.1 로그 분석

보안 로그는 순서가 중요하다.

```text
로그인 실패 → 권한 상승 시도 → 의심 프로세스 실행 → 외부 IP 통신
```

이런 이벤트 시퀀스는 과거 RNN/LSTM 계열 모델이 잘 다루려 했던 문제와 비슷하다.

즉, 시간 순서가 중요한 데이터는 sequence modeling 관점으로 볼 수 있다.

### 20.2 RAG

RAG에서는 문서를 chunk로 나누고, 필요한 chunk를 검색해 LLM에 넣는다.

여기서도 긴 문맥을 어떻게 다룰지가 핵심이다.

RNN/LSTM은 긴 문장을 하나의 상태에 압축하려 했고, Seq2Seq는 입력 문장을 하나의 Context Vector에 압축하려 했다.

RAG는 다르게 접근한다.

```text
모든 문서를 모델 내부에 기억시키는 대신,
필요한 문서 조각을 외부에서 검색해 context로 넣는다.
```

즉, 긴 정보 처리 문제를 모델 내부 구조만이 아니라 검색 시스템과 결합해서 해결하는 방식이다.

### 20.3 Agent

Agent는 대화 흐름과 tool 사용 이력을 관리한다.

이것도 일종의 sequence다.

```text
사용자 질문 → 도구 선택 → 도구 실행 결과 → 재판단 → 최종 응답
```

이 흐름을 이해하려면 모델이 이전 상태를 어떻게 반영하는지에 대한 감각이 필요하다.

RNN의 hidden state, LSTM의 memory, Transformer의 context window는 모두 이 문제와 연결된다.

## 21. 꼭 기억해야 할 핵심 정리

이번 편에서 꼭 기억해야 할 내용은 다음과 같다.

```text
Word2Vec은 단어 의미를 벡터로 표현했지만, 문장 순서와 긴 문맥 처리에는 한계가 있었다.

RNN은 문장을 순서대로 읽으며 hidden state에 이전 정보를 저장하는 구조다.

RNN은 순서 정보를 다룰 수 있지만, 긴 문장에서 앞쪽 정보를 잘 기억하지 못하는 장기 의존성 문제가 있었다.

LSTM은 gate 구조를 사용해 중요한 정보는 유지하고 불필요한 정보는 버리는 방식으로 RNN의 한계를 완화했다.

Seq2Seq는 Encoder-Decoder 구조를 통해 입력 시퀀스를 출력 시퀀스로 변환하는 생성형 NLP의 중요한 기반이 되었다.

하지만 Seq2Seq는 입력 문장을 하나의 Context Vector에 압축해야 해서 긴 문장에서 정보 병목이 발생했다.

이 문제를 해결하기 위해 Attention이 등장했다.

Attention은 이후 Transformer와 LLM의 핵심 기반이 된다.
```
