# LLM 공부 9편: Pretraining과 Next Token Prediction

## 0. 이번 편의 목표

이번 편에서는 LLM이 어떻게 처음 학습되는지 살펴본다.

지금까지는 다음 흐름을 봤다.

```text
1편: LLM의 전체 발전 흐름
2편: Word2Vec과 단어 임베딩
3편: RNN, LSTM, Seq2Seq
4편: Attention
5편: Transformer 구조
6편: Self-Attention과 Multi-Head Attention
7편: GPT와 Decoder-only Transformer
8편: Tokenization과 Embedding
```

이번 9편의 핵심은 이것이다.

```text
LLM은 처음부터 질문에 답하도록 학습되는 것이 아니다.
먼저 대규모 텍스트에서 다음 토큰을 맞히는 방식으로 학습된다.
```

이 과정을 보통 **Pretraining**, 즉 사전학습이라고 부른다.

---

## 1. Pretraining이란?

Pretraining은 모델이 특정 업무를 수행하기 전에, 대규모 데이터로 기본적인 언어 능력을 학습하는 과정이다.

예를 들어 GPT 계열 모델은 웹 문서, 책, 코드, 위키, 논문, 대화 데이터 등 매우 많은 텍스트를 보고 다음 토큰을 예측하도록 학습된다.

단순화하면 이런 문제를 계속 푸는 것이다.

```text
입력:
오늘 날씨가 정말

정답:
좋다
```

또 다른 예시는 이렇다.

```text
입력:
Python에서 리스트의 길이를 구할 때는

정답:
len
```

모델은 이런 식으로 수많은 문장에서 다음에 올 토큰을 맞히는 연습을 한다.

중요한 점은 이 단계에서 모델이 반드시 “챗봇처럼 친절하게 답변하는 법”을 배우는 것은 아니라는 점이다.

Pretraining의 목표는 더 넓다.

```text
언어 패턴 학습
문법 구조 학습
단어와 개념 간 관계 학습
사실 정보 일부 학습
코드 패턴 학습
추론에 자주 쓰이는 문장 패턴 학습
도메인별 표현 방식 학습
```

즉, Pretraining은 LLM의 기본 체력을 만드는 단계다.

## 2. Next Token Prediction이란?

GPT 계열 LLM의 가장 기본적인 학습 목표는 **Next Token Prediction**이다.

말 그대로 다음 토큰을 예측하는 것이다.

예를 들어 문장이 있다고 하자.

```text
보안 관제 시스템에서는 이벤트 로그를 분석한다.
```

토큰화된 결과를 단순화하면 다음과 같다고 볼 수 있다.

```text
[보안, 관제, 시스템, 에서는, 이벤트, 로그, 를, 분석, 한다, .]
```

모델은 다음과 같은 학습 문제를 반복해서 푼다.

```text
보안 → 관제
보안 관제 → 시스템
보안 관제 시스템 → 에서는
보안 관제 시스템에서는 → 이벤트
보안 관제 시스템에서는 이벤트 → 로그
```

즉, 이전 토큰들을 보고 다음 토큰을 맞히는 방식이다.

이것을 조금 더 일반화하면 다음과 같다.

```text
P(next_token | previous_tokens)
```

읽으면 이렇게 된다.

```text
이전 토큰들이 주어졌을 때, 다음 토큰이 무엇일 확률이 높은가?
```

LLM의 생성 과정도 결국 이 원리와 같다.

```text
1. 사용자의 입력을 토큰으로 변환한다.
2. 모델이 다음 토큰의 확률 분포를 계산한다.
3. 확률이 높은 토큰 중 하나를 선택한다.
4. 선택된 토큰을 입력 뒤에 붙인다.
5. 다시 다음 토큰을 예측한다.
6. 종료 조건이 나올 때까지 반복한다.
```

그래서 LLM은 한 번에 긴 답변을 쓰는 것이 아니다.  
**다음 토큰을 하나씩 생성하는 과정을 반복하면서 답변을 만든다.**

---

## 3. 예측한다는 것은 무엇인가?

모델이 다음 토큰을 예측한다는 것은 단순히 하나의 단어만 출력한다는 뜻이 아니다.

실제로는 전체 vocabulary에 대해 확률을 계산한다.

예를 들어 입력이 다음과 같다고 하자.

```text
대한민국의 수도는
```

모델은 다음 후보들에 대해 확률을 계산한다.

| 후보 토큰 | 확률 예시 |
|---|---:|
| 서울 | 0.82 |
| 부산 | 0.04 |
| 한국 | 0.03 |
| 도시 | 0.02 |
| 일본 | 0.01 |
| 기타 | 0.08 |

모델 입장에서는 다음 토큰이 “서울”일 가능성이 가장 높다고 판단하는 것이다.

이 확률 분포를 기반으로 실제 출력 토큰을 고른다.

이때 항상 가장 높은 확률의 토큰만 고르면 답변이 단조로워질 수 있다.  
반대로 너무 무작위로 고르면 이상한 답변이 나올 수 있다.

그래서 temperature, top-k, top-p 같은 sampling 기법이 필요해진다.

이 내용은 이후 편에서 따로 다루는 것이 좋다.

---

## 4. 왜 다음 토큰 예측만으로 언어 능력이 생길까?

처음 보면 이상하게 느껴질 수 있다.

```text
“그냥 다음 단어 맞히기만 했는데 왜 번역, 요약, 코딩, 추론까지 하지?”
```

핵심은 데이터의 규모와 모델의 규모다.

다음 토큰을 잘 맞히려면 모델은 단순히 앞 단어만 보면 안 된다.  
문맥, 문법, 개념 관계, 배경 지식, 문서 구조를 모두 활용해야 한다.

예를 들어 다음 문장을 보자.

```text
취약점 CVE-2021-44228은 Apache Log4j와 관련된 심각한
```

이 문장에서 다음 토큰을 잘 맞히려면 모델은 다음 정보를 어느 정도 알고 있어야 한다.

```text
CVE-2021-44228이 Log4Shell과 관련 있다는 것
Apache Log4j가 Java 로깅 라이브러리라는 것
이 취약점이 원격 코드 실행과 관련 있다는 것
보안 문서에서 보통 “취약점”, “영향”, “완화”, “패치” 같은 표현이 이어진다는 것
```

즉, 다음 토큰을 잘 예측하는 문제는 생각보다 어렵다.

모델이 매우 많은 데이터를 보고 이 문제를 반복해서 풀면, 내부 파라미터에는 다양한 패턴이 압축된다.

이것이 LLM의 기본 능력이 된다.

## 5. Pretraining 데이터는 왜 중요한가?

LLM의 성능은 모델 구조만으로 결정되지 않는다.

데이터가 매우 중요하다.

좋은 Pretraining 데이터에는 보통 다음 특성이 필요하다.

```text
양이 충분히 많아야 한다.
도메인이 다양해야 한다.
중복이 너무 많으면 안 된다.
품질이 낮은 텍스트가 과도하면 안 된다.
유해하거나 잘못된 정보가 너무 많으면 문제가 된다.
코드, 수학, 논문, 대화, 문서 등 다양한 형식이 섞이면 좋다.
```

모델은 데이터를 보고 학습하기 때문에 데이터에 포함된 편향, 오류, 표현 방식도 함께 배운다.

예를 들어 보안 도메인 LLM을 만든다고 생각해보자.

일반 웹 텍스트만 학습한 모델보다 다음 데이터를 포함한 모델이 보안 업무에 더 유리할 수 있다.

```text
CVE 설명
NVD 문서
MITRE ATT&CK 문서
보안 블로그
침해사고 분석 보고서
IOC 피드
악성코드 분석 리포트
탐지 룰
로그 분석 예시
```

하지만 단순히 많이 넣는다고 좋은 것은 아니다.  
중복 제거, 품질 필터링, 개인정보 제거, 라이선스 검토 같은 작업도 필요하다.

---

## 6. 학습 데이터는 어떻게 만들어질까?

Pretraining 데이터 준비 과정은 보통 다음과 같다.

```text
원천 데이터 수집
→ 텍스트 추출
→ 언어 감지
→ 품질 필터링
→ 중복 제거
→ 개인정보/민감정보 제거
→ 토큰화
→ 학습 샘플 구성
```

예를 들어 PDF, HTML, Markdown, 코드 파일, JSON 문서 등 다양한 형식의 데이터를 수집하면 그대로 학습에 쓰기 어렵다.

HTML이면 태그를 제거해야 하고, PDF면 텍스트를 추출해야 하며, 코드 파일이면 파일 경로와 언어 정보를 함께 보존할 수도 있다.

네가 RAG용 문서를 chunking할 때도 비슷한 문제가 생긴다.

```text
문서를 어떻게 자를 것인가?
메타데이터를 어디까지 보존할 것인가?
표와 코드 블록은 어떻게 처리할 것인가?
중복 문단은 제거할 것인가?
본문만 임베딩할 것인가?
```

Pretraining 규모에서는 이 작업이 훨씬 더 크고 복잡하게 진행된다고 보면 된다.

## 7. 입력 길이와 학습 샘플

LLM은 한 번에 무한히 긴 문서를 볼 수 없다.

모델마다 context window가 있다.

예를 들어 context window가 2048 tokens인 모델이라면, 학습 시에도 보통 일정 길이의 토큰 블록으로 잘라서 학습한다.

단순화하면 다음과 같다.

```text
원본 텍스트:
토큰 1, 토큰 2, 토큰 3, ..., 토큰 100000

학습 샘플:
[토큰 1 ~ 토큰 2048]
[토큰 2049 ~ 토큰 4096]
[토큰 4097 ~ 토큰 6144]
...
```

각 샘플 안에서 모델은 다음 토큰 예측을 수행한다.

여기서 중요한 점은 context window 안에 들어오지 않은 정보는 모델이 직접 참조할 수 없다는 것이다.

이 한계는 RAG가 필요한 이유와도 연결된다.

```text
모델 파라미터에 저장된 정보만으로 답변하기 어렵다.
최신 정보는 사전학습 시점 이후에는 알 수 없다.
긴 사내 문서 전체를 한 번에 넣을 수 없다.
그래서 검색을 통해 필요한 문서를 context에 넣어주는 RAG가 필요하다.
```

## 8. Loss는 어떻게 계산할까?

Next Token Prediction에서는 모델이 예측한 확률과 실제 정답 토큰을 비교한다.

예를 들어 정답 토큰이 “서울”인데, 모델이 다음과 같이 예측했다고 하자.

| 후보 토큰 | 예측 확률 |
|---|---:|
| 서울 | 0.82 |
| 부산 | 0.05 |
| 도쿄 | 0.03 |
| 기타 | 0.10 |

정답인 “서울”의 확률이 높기 때문에 loss는 낮다.

반대로 모델이 이렇게 예측했다면 어떨까?

| 후보 토큰 | 예측 확률 |
|---|---:|
| 서울 | 0.05 |
| 부산 | 0.70 |
| 도쿄 | 0.10 |
| 기타 | 0.15 |

정답인 “서울”의 확률이 낮기 때문에 loss는 높다.

모델은 이 loss를 줄이는 방향으로 파라미터를 업데이트한다.

일반적으로 언어 모델 학습에서는 **cross entropy loss**를 사용한다.

학습 목표는 간단히 말하면 이것이다.

```text
정답 토큰의 확률은 높이고,
틀린 토큰의 확률은 낮추는 방향으로 모델을 조정한다.
```

## 9. Causal Masking과 Pretraining의 관계

GPT 같은 Decoder-only Transformer는 왼쪽에서 오른쪽으로 텍스트를 생성한다.

따라서 학습할 때도 미래 토큰을 보면 안 된다.

예를 들어 다음 문장이 있다고 하자.

```text
나는 오늘 보안 로그를 분석했다.
```

“보안”이라는 토큰을 예측할 때 모델이 뒤에 있는 “로그를 분석했다”를 미리 보면 안 된다.  
그렇게 되면 정답을 훔쳐보는 것이 된다.

그래서 GPT 계열 모델은 **causal masking**을 사용한다.

Causal masking은 각 토큰이 자기보다 오른쪽에 있는 미래 토큰을 보지 못하게 막는 장치다.

단순화하면 다음과 같다.

```text
토큰 1은 토큰 1까지만 볼 수 있다.
토큰 2는 토큰 1~2까지만 볼 수 있다.
토큰 3은 토큰 1~3까지만 볼 수 있다.
토큰 4는 토큰 1~4까지만 볼 수 있다.
```

이 덕분에 GPT는 생성 시점과 동일한 조건으로 학습된다.

```text
학습 때도 과거 토큰만 보고 다음 토큰 예측
추론 때도 과거 토큰만 보고 다음 토큰 생성
```

이 구조가 GPT 계열 모델의 핵심이다.

## 10. Pretraining과 Fine-tuning의 차이

Pretraining과 Fine-tuning은 목적이 다르다.

| 구분 | Pretraining | Fine-tuning |
|---|---|---|
| 목적 | 기본 언어 능력 학습 | 특정 목적에 맞게 조정 |
| 데이터 | 대규모 일반 텍스트 | 상대적으로 작은 고품질 데이터 |
| 학습 방식 | 주로 다음 토큰 예측 | 지시문-응답, 도메인 데이터, 선호 데이터 등 |
| 결과 | Base model | Instruct model, Chat model, Domain model |
| 비용 | 매우 큼 | 상대적으로 작음 |

Base model은 다음 텍스트를 이어 쓰는 능력이 강하다.  
하지만 사용자의 질문에 친절하고 안전하게 답변하는 능력은 부족할 수 있다.

예를 들어 사용자가 이렇게 물었다고 하자.

```text
CVE-2021-44228에 대해 설명해줘.
```

Base model은 문서를 이어 쓰듯이 답할 수 있다.  
반면 Instruct model은 질문에 대한 답변 형식으로 정리해서 말한다.

이 차이는 Fine-tuning과 Alignment 단계에서 만들어진다.

## 11. Base Model은 왜 그대로 쓰기 어려울까?

Base model은 대규모 텍스트를 이어 쓰도록 학습된 모델이다.

그래서 이런 문제가 생길 수 있다.

```text
질문에 직접 답하지 않고 문서를 이어 쓴다.
사용자 의도를 잘 따르지 못한다.
대화 형식을 안정적으로 유지하지 못한다.
안전하지 않은 답변을 할 수 있다.
답변 품질이 들쭉날쭉하다.
```

예를 들어 사용자가 이렇게 입력했다고 하자.

```text
다음 SQL 쿼리를 고쳐줘:
SELECT * FORM users;
```

Base model은 실제 수정 답변을 주기보다, 비슷한 코드 예시나 튜토리얼 문장을 이어 쓸 수 있다.

그래서 실제 챗봇 서비스에는 보통 base model을 그대로 쓰지 않는다.

일반적으로는 다음 과정을 거친다.

```text
Pretraining
→ Supervised Fine-Tuning
→ Preference Alignment
→ Safety Tuning
→ Chat/Instruction Model
```

이후 편에서 SFT, RLHF, DPO를 따로 다룰 것이다.

## 12. “지식을 학습한다”는 표현의 한계

LLM을 설명할 때 흔히 “모델이 지식을 학습한다”라고 말한다.

틀린 말은 아니지만 정확하게는 조금 조심해야 한다.

LLM은 데이터베이스처럼 지식을 행 단위로 저장하지 않는다.

예를 들어 PostgreSQL 테이블처럼 다음 구조로 저장하는 것이 아니다.

```text
key: CVE-2021-44228
value: Log4Shell, Apache Log4j RCE vulnerability
```

대신 모델의 수많은 파라미터에 통계적 패턴이 분산되어 저장된다.

그래서 다음과 같은 특성이 있다.

```text
자주 등장한 정보는 비교적 잘 기억할 수 있다.
희귀한 정보는 부정확할 수 있다.
최신 정보는 알 수 없다.
비슷한 개념을 섞어 말할 수 있다.
출처를 정확히 추적하기 어렵다.
```

이것이 hallucination이 발생하는 큰 이유 중 하나다.

그래서 실무 시스템에서는 다음 방식을 함께 사용한다.

```text
RAG로 최신 문서 검색
Tool Calling으로 외부 시스템 조회
SQL Agent로 실제 DB 조회
출처와 근거를 함께 제공
LLM 답변 후 검증 단계 추가
```

네가 구축 중인 MDR multi-agent 시스템도 결국 이 방향과 잘 맞는다.

## 13. Pretraining 관점에서 RAG를 이해하기

Pretraining은 모델의 기본 지식을 만든다.  
하지만 Pretraining만으로 실무 시스템을 만들기에는 한계가 있다.

예를 들어 보안 관제 챗봇이 있다고 하자.

사용자가 이렇게 묻는다.

```text
최근 7일 동안 발생한 high severity incident를 정리해줘.
```

이 질문은 Pretraining으로 답할 수 없다.

왜냐하면 최근 7일의 incident 정보는 모델 파라미터 안에 없기 때문이다.

이때 필요한 것이 외부 데이터 조회다.

```text
Cortex XDR MCP tool 호출
TI PostgreSQL 조회
보안 문서 RAG 검색
실시간 로그 조회
```

LLM은 Pretraining으로 언어 능력과 일반적인 보안 지식을 갖고,  
실제 최신 데이터는 RAG나 Tool Calling으로 가져온다.

정리하면 다음과 같다.

```text
Pretraining = 모델의 기본 언어 능력과 일반 지식
RAG = 외부 문서 기반 근거 보강
Tool Calling = 실시간 시스템 조회
Agent = 여러 도구와 판단 흐름을 조합
```

## 14. 작은 예제로 이해하기

다음 문장이 있다고 하자.

```text
침해사고 대응에서 가장 먼저 해야 할 일은
```

사전학습된 모델은 다음 후보를 예측할 수 있다.

```text
확인
분석
격리
로그
탐지
```

보안 문서를 많이 본 모델이라면 “초기 식별”, “영향 범위 확인”, “격리” 같은 표현을 더 자연스럽게 예측할 수 있다.

하지만 사용자가 이렇게 물으면 이야기가 달라진다.

```text
우리 회사의 어제 침해사고 대응 상황을 요약해줘.
```

이 경우 모델은 사전학습만으로 답하면 안 된다.  
실제 회사의 어제 로그, 티켓, 사고 보고서를 조회해야 한다.

즉, Pretraining은 답변 능력의 기반이지만, 실시간 사실 확인까지 보장하지는 않는다.

## 15. 실습 아이디어

이번 편을 이해했다면 다음 실습을 해보면 좋다.

### 실습 1: 작은 GPT 모델로 다음 토큰 확인하기

Hugging Face의 작은 모델을 사용해서 입력 문장의 다음 토큰 후보를 확인해볼 수 있다.

예시 모델:

```text
gpt2
distilgpt2
Qwen/Qwen2.5-0.5B
```

확인해볼 문장:

```text
The capital city of Korea is
Python list length can be calculated using
A severe vulnerability in Log4j is called
```

보고 싶은 것:

```text
다음 토큰 후보가 무엇인지
각 후보의 확률이 어떻게 다른지
입력 문장을 바꾸면 후보가 어떻게 변하는지
```

### 실습 2: Temperature에 따른 출력 차이 보기

같은 prompt에 대해 temperature를 바꿔가며 결과를 비교한다.

```text
temperature = 0.0
temperature = 0.3
temperature = 0.7
temperature = 1.0
```

낮은 temperature는 더 안정적이고, 높은 temperature는 더 다양하지만 불안정해질 수 있다.

### 실습 3: 보안 도메인 문장으로 테스트하기

다음과 같은 문장을 넣어본다.

```text
CVE-2021-44228 is a vulnerability related to
MITRE ATT&CK technique T1059 refers to
In incident response, the first step is to
```

일반 모델이 보안 도메인 표현을 얼마나 잘 이어 쓰는지 확인해보면 된다.

## 16. 자주 헷갈리는 포인트

### 16.1 LLM은 검색 엔진인가?

아니다.

LLM은 기본적으로 다음 토큰을 예측하는 생성 모델이다.  
검색 엔진처럼 문서를 찾아서 그대로 보여주는 시스템이 아니다.

다만 RAG와 결합하면 검색 엔진처럼 외부 문서를 참조할 수 있다.

### 16.2 Pretraining만 하면 ChatGPT 같은 모델이 되는가?

아니다.

Pretraining은 base model을 만드는 단계다.

ChatGPT 같은 assistant 모델이 되려면 instruction tuning, preference alignment, safety tuning 등이 필요하다.

### 16.3 모델은 학습 데이터를 그대로 외우는가?

일부는 외울 수 있다.  
특히 중복되거나 자주 등장하는 문장은 memorization이 발생할 수 있다.

하지만 일반적으로는 데이터를 DB처럼 저장하는 것이 아니라, 파라미터에 패턴을 분산 저장한다.

### 16.4 최신 정보를 알고 있는가?

Pretraining 시점 이후의 정보는 기본적으로 알 수 없다.

최신 정보가 필요하면 다음이 필요하다.

```text
검색
RAG
DB 조회
API 호출
Tool Calling
```

### 16.5 다음 토큰 예측이 곧 추론인가?

완전히 같은 말은 아니다.

LLM의 추론 능력은 다음 토큰 예측 학습 과정에서 형성된 패턴, 대규모 데이터, 모델 크기, instruction tuning, reasoning 데이터 등에 의해 나타난다.

하지만 내부적으로 생성은 여전히 다음 토큰 예측의 반복이다.

## 17. 이번 편 핵심 요약

이번 편의 핵심은 다음과 같다.

```text
Pretraining은 LLM의 기본 체력을 만드는 단계다.
GPT 계열 모델은 주로 Next Token Prediction으로 학습된다.
다음 토큰을 잘 맞히려면 문법, 의미, 지식, 문맥을 모두 활용해야 한다.
모델은 전체 vocabulary에 대해 다음 토큰 확률을 계산한다.
Causal masking 덕분에 미래 토큰을 보지 않고 학습한다.
Base model은 그대로 챗봇처럼 쓰기 어렵다.
Chat model이 되려면 SFT, RLHF, DPO 같은 후속 학습이 필요하다.
Pretraining만으로 최신 정보나 사내 데이터까지 정확히 답할 수는 없다.
그래서 RAG, Tool Calling, Agent 구조가 필요하다.
```

한 문장으로 정리하면 이렇다.

```text
LLM은 대규모 텍스트에서 다음 토큰을 맞히는 방식으로 언어와 지식의 패턴을 압축하고, 이후 instruction tuning과 alignment를 통해 우리가 쓰는 assistant 형태로 다듬어진다.
```

## 18. 다음 편 예고

다음 10편에서는 Pretraining 이후 단계인 **Supervised Fine-Tuning, Instruction Tuning, RLHF, DPO**를 다루는 것이 좋다.

다음 질문들을 중심으로 보면 된다.

```text
Base model과 Instruct model은 무엇이 다른가?
SFT는 어떤 데이터로 학습하는가?
Instruction tuning은 왜 필요한가?
RLHF는 왜 등장했는가?
DPO는 RLHF와 무엇이 다른가?
Alignment는 정확히 무엇을 맞춘다는 뜻인가?
```

이 부분을 이해하면 “모델을 학습시킨다”는 말이 훨씬 구체적으로 보이기 시작한다.
