# LLM 공부 1편: LLM의 시초와 전체 발전 흐름

## 0. 이번 편의 목표

이번 편에서는 LLM을 깊게 공부하기 전에, 먼저 **LLM이 어디서부터 시작되었고 어떤 흐름을 거쳐 지금의 형태가 되었는지**를 잡는다.

아직 수식이나 구현 코드는 최소화한다.  
목표는 아래 질문에 답할 수 있게 되는 것이다.

```text
LLM은 갑자기 등장한 기술인가?
Word2Vec, RNN, LSTM, Attention, Transformer는 각각 왜 등장했는가?
GPT와 BERT는 어떤 흐름에서 나왔는가?
ChatGPT 같은 모델은 기존 GPT와 무엇이 다른가?
```

---

## 1. LLM은 갑자기 나온 기술이 아니다

요즘은 LLM이라고 하면 GPT, Claude, Gemini, Llama, Qwen 같은 모델을 떠올린다.

하지만 LLM은 갑자기 등장한 기술이 아니다.  
오랫동안 이어진 자연어 처리, 딥러닝, 대규모 데이터, GPU 연산, 분산 학습, 모델 최적화 기술이 쌓이면서 만들어진 결과물이다.

큰 흐름은 다음과 같다.

```text
규칙 기반 NLP
→ 통계 기반 NLP
→ 단어 임베딩
→ RNN / LSTM
→ Seq2Seq
→ Attention
→ Transformer
→ BERT / GPT
→ Instruction-tuned LLM
→ RAG / Agent / Tool-use / Multimodal LLM
```

이 흐름을 먼저 알아야 한다.  
왜냐하면 LLM의 핵심 구조와 한계가 대부분 이 역사 안에서 설명되기 때문이다.

## 2. 규칙 기반 NLP: 사람이 언어 규칙을 직접 만들던 시기

초기 자연어 처리에서는 사람이 직접 규칙을 만들었다.

예를 들어 다음과 같은 방식이다.

```text
문장에 "위험", "공격", "악성"이라는 단어가 있으면 보안 위협으로 분류한다.
동사 앞에 특정 조사가 있으면 특정 문장 구조로 판단한다.
정규표현식으로 이메일, IP, 도메인, URL을 추출한다.
```

이 방식은 단순한 문제에는 잘 동작한다.

예를 들어 보안 로그에서 IP를 추출하는 작업은 정규표현식으로도 충분히 가능하다.

```regex
\b(?:\d{1,3}\.){3}\d{1,3}\b
```

하지만 언어는 너무 복잡하다.

같은 의미를 여러 방식으로 표현할 수 있고, 문맥에 따라 단어의 의미가 달라진다.

예를 들어 아래 문장들을 보자.

```text
이 파일은 악성으로 판단됩니다.
이 파일은 악성일 가능성이 낮습니다.
악성으로 오탐된 파일입니다.
악성 행위는 발견되지 않았습니다.
```

단순히 `악성`이라는 단어만 보면 모두 위험해 보이지만, 실제 의미는 다르다.

그래서 규칙 기반 방식은 한계가 명확했다.

### 핵심 한계

```text
사람이 규칙을 계속 만들어야 한다.
예외 처리가 너무 많아진다.
문맥 이해가 어렵다.
새로운 표현에 약하다.
언어가 복잡해질수록 유지보수가 어렵다.
```

## 3. 통계 기반 NLP: 언어를 확률로 보기 시작한 시기

규칙 기반 방식의 한계를 넘기 위해 언어를 확률적으로 다루기 시작했다.

예를 들어 이런 질문을 던진다.

```text
어떤 단어 다음에 어떤 단어가 나올 확률이 높은가?
어떤 문장이 더 자연스러운가?
어떤 단어 조합이 특정 주제와 자주 함께 등장하는가?
```

대표적인 방식이 N-gram 모델이다.

N-gram은 연속된 N개의 단어 묶음을 이용한다.

예를 들어 문장이 다음과 같다고 하자.

```text
사용자가 악성 파일을 실행했다
```

2-gram으로 보면 다음과 같다.

```text
사용자가 악성
악성 파일을
파일을 실행했다
```

3-gram으로 보면 다음과 같다.

```text
사용자가 악성 파일을
악성 파일을 실행했다
```

이 방식은 단어의 연속 패턴을 통계적으로 학습한다.

예를 들어 `악성` 다음에는 `파일`, `코드`, `행위`, `트래픽` 같은 단어가 자주 나올 수 있다.

### 장점

```text
사람이 모든 규칙을 직접 만들 필요가 줄어든다.
데이터를 기반으로 언어 패턴을 학습할 수 있다.
문장 생성, 자동완성, 간단한 분류에 활용할 수 있다.
```

### 한계

하지만 N-gram도 한계가 크다.

```text
긴 문맥을 이해하기 어렵다.
단어의 의미를 제대로 이해하지 못한다.
비슷한 의미의 단어를 비슷하게 다루지 못한다.
데이터에 없던 표현에 약하다.
차원이 커지고 희소성이 심해진다.
```

예를 들어 아래 두 문장은 의미가 비슷하다.

```text
공격자가 서버에 침입했다.
해커가 시스템에 무단 접근했다.
```

사람은 비슷한 의미라고 이해할 수 있다.

하지만 단어 기반 통계 모델은 `공격자`, `해커`, `서버`, `시스템`, `침입`, `무단 접근`이 서로 의미적으로 가깝다는 것을 자연스럽게 알기 어렵다.

이 문제를 해결하기 위해 등장한 것이 **단어 임베딩**이다.

## 4. 단어 임베딩: 단어를 벡터로 표현하기

LLM으로 가는 중요한 첫 번째 전환점은 **단어를 숫자 벡터로 표현하는 것**이다.

컴퓨터는 텍스트를 그대로 이해하지 못한다.  
결국 숫자로 바꿔야 한다.

초기에는 단어를 one-hot vector로 표현했다.

예를 들어 단어 사전이 다음과 같다고 하자.

```text
[공격자, 서버, 파일, 악성, 정상]
```

그러면 각 단어는 이렇게 표현될 수 있다.

```text
공격자 = [1, 0, 0, 0, 0]
서버   = [0, 1, 0, 0, 0]
파일   = [0, 0, 1, 0, 0]
악성   = [0, 0, 0, 1, 0]
정상   = [0, 0, 0, 0, 1]
```

문제는 이 방식에서는 단어 간 의미적 관계가 없다.

`공격자`와 `해커`가 비슷한 의미여도 one-hot vector에서는 둘의 관계를 알 수 없다.

그래서 등장한 방식이 Word2Vec, GloVe 같은 단어 임베딩이다.

단어 임베딩에서는 단어를 고차원 벡터로 표현한다.

```text
공격자 = [0.12, -0.44, 0.91, ...]
해커   = [0.10, -0.39, 0.87, ...]
서버   = [-0.21, 0.33, 0.15, ...]
```

비슷한 문맥에서 자주 등장하는 단어는 벡터 공간에서도 가까워진다.

즉, 단어의 의미를 벡터 공간에 배치하는 것이다.

네가 pgvector나 vector DB에서 하는 유사도 검색도 이 아이디어와 연결된다.

```text
텍스트 → 임베딩 벡터 → 벡터 공간에서 가까운 항목 검색
```

### 단어 임베딩이 중요한 이유

```text
단어를 숫자 벡터로 표현할 수 있게 되었다.
비슷한 의미의 단어를 비슷한 벡터로 표현할 수 있게 되었다.
검색, 분류, 추천, 문서 유사도 계산의 기반이 되었다.
현대 LLM의 embedding layer로 이어졌다.
```

하지만 단어 임베딩에도 한계가 있었다.

가장 큰 문제는 **문맥에 따라 단어 의미가 달라지는 것**을 충분히 반영하기 어렵다는 점이다.

예를 들어 `bank`는 문맥에 따라 은행일 수도 있고 강둑일 수도 있다.

한국어에서도 마찬가지다.

```text
파일을 실행했다.
계획을 실행했다.
명령어를 실행했다.
```

여기서 `실행`은 비슷하지만 문맥마다 조금씩 의미가 다르다.

고정된 단어 벡터만으로는 이런 문맥 차이를 표현하기 어렵다.

## 5. RNN과 LSTM: 문장을 순서대로 읽는 모델

단어 임베딩 이후에는 문장의 순서를 처리하는 모델이 중요해졌다.

언어는 순서가 중요하다.

```text
개가 사람을 물었다.
사람이 개를 물었다.
```

같은 단어가 등장해도 순서가 바뀌면 의미가 완전히 달라진다.

이런 순서 정보를 처리하기 위해 RNN이 사용되었다.

RNN은 문장을 왼쪽에서 오른쪽으로 순서대로 읽는다.

```text
사용자가 → 악성 → 파일을 → 실행했다
```

각 단어를 읽으면서 이전 상태를 다음 상태로 넘긴다.

```text
h1 = f(사용자가)
h2 = f(h1, 악성)
h3 = f(h2, 파일을)
h4 = f(h3, 실행했다)
```

이 구조는 문맥을 누적해서 다룰 수 있다는 장점이 있다.

하지만 큰 문제가 있었다.

긴 문장이 되면 앞부분 정보가 뒤로 갈수록 약해진다.

이를 장기 의존성 문제라고 한다.

예를 들어 다음 문장을 보자.

```text
어제 새벽부터 여러 차례 로그인 실패가 발생했고, 이후 관리자 계정으로 정상 로그인된 뒤, 내부 DB 서버에서 대량의 데이터가 외부 IP로 전송되었다.
```

이 문장을 분석하려면 앞의 `로그인 실패`, 중간의 `관리자 계정 로그인`, 뒤의 `데이터 외부 전송`을 모두 연결해야 한다.

RNN은 이런 긴 문맥을 잘 유지하기 어렵다.

이 문제를 개선하기 위해 LSTM과 GRU가 등장했다.

### LSTM의 핵심 아이디어

LSTM은 중요한 정보를 오래 기억하고, 덜 중요한 정보는 잊도록 설계된 RNN 계열 모델이다.

```text
무엇을 기억할지
무엇을 잊을지
무엇을 다음 상태로 넘길지
```

이런 제어 구조를 통해 RNN보다 긴 문맥을 더 잘 처리할 수 있었다.

하지만 LSTM도 완벽하지 않았다.

### RNN/LSTM의 한계

```text
문장을 순서대로 처리하기 때문에 병렬화가 어렵다.
긴 문맥을 완전히 해결하지 못한다.
멀리 떨어진 단어 간 관계를 직접 연결하기 어렵다.
대규모 학습에 비효율적이다.
```

이 한계가 다음 단계인 Attention으로 이어진다.

## 6. Attention: 중요한 단어를 직접 바라보게 만들기

Attention의 핵심 아이디어는 단순하다.

```text
모든 단어를 똑같이 보지 말고, 현재 처리에 중요한 단어에 더 집중하자.
```

예를 들어 번역 작업을 생각해보자.

```text
The attacker accessed the server.
공격자가 서버에 접근했다.
```

`accessed`를 번역할 때는 `attacker`, `server`와의 관계가 중요하다.

Attention은 현재 단어를 처리할 때 입력 문장의 어떤 단어를 더 중요하게 볼지 가중치를 계산한다.

이 방식 덕분에 멀리 떨어진 단어 간 관계도 더 잘 다룰 수 있게 되었다.

예를 들어 다음 문장에서:

```text
관리자 계정으로 로그인한 사용자가 내부 DB 서버에서 대량의 데이터를 외부로 전송했다.
```

`전송했다`를 이해할 때 중요한 단어는 다음과 같다.

```text
관리자 계정
내부 DB 서버
대량의 데이터
외부
```

Attention은 이런 관련 단어들을 직접 연결할 수 있게 해준다.

### Attention이 중요한 이유

```text
긴 문장에서 중요한 단어를 직접 참고할 수 있다.
RNN/LSTM의 장기 의존성 문제를 완화했다.
번역, 요약, 질의응답 성능을 크게 끌어올렸다.
Transformer의 핵심 구성 요소가 되었다.
```

## 7. Transformer: LLM 시대를 연 구조

2017년에 등장한 Transformer는 LLM 역사에서 가장 중요한 전환점이다.

Transformer의 핵심은 다음과 같다.

```text
RNN 없이 Attention만으로 문장을 처리하자.
```

RNN은 문장을 순서대로 처리한다.

```text
1번째 토큰 → 2번째 토큰 → 3번째 토큰 → 4번째 토큰
```

반면 Transformer는 Self-Attention을 사용해서 토큰 간 관계를 한 번에 계산할 수 있다.

이 덕분에 병렬 처리가 훨씬 쉬워졌다.

병렬 처리가 가능해졌다는 것은 매우 중요하다.

왜냐하면 대규모 데이터와 GPU를 사용해서 훨씬 큰 모델을 학습할 수 있게 되었기 때문이다.

### Transformer의 핵심 구성요소

```text
Tokenization
Embedding
Positional Encoding
Self-Attention
Multi-Head Attention
Feed Forward Network
Residual Connection
Layer Normalization
```

지금은 이름만 익숙해지면 된다.

다음 편들에서 하나씩 자세히 볼 것이다.

### Transformer가 중요한 이유

```text
RNN보다 병렬 학습에 유리하다.
긴 문맥에서 토큰 간 관계를 더 잘 다룬다.
모델 크기를 키우기 좋다.
BERT, GPT, T5, Llama 등 현대 LLM의 기반이 되었다.
```

## 8. BERT와 GPT: Transformer 이후의 두 큰 흐름

Transformer 이후 NLP 모델은 크게 두 흐름으로 발전했다.

```text
Encoder 중심 모델: BERT 계열
Decoder 중심 모델: GPT 계열
```

### BERT

BERT는 문장을 이해하는 데 강한 모델이다.

주요 특징은 다음과 같다.

```text
Encoder-only Transformer
양방향 문맥 이해
Masked Language Modeling
분류, 검색, 문장 유사도, NER 등에 강함
```

예를 들어 문장 중간의 단어를 가리고 맞히는 방식으로 학습한다.

```text
사용자가 [MASK] 파일을 실행했다.
```

모델은 문맥을 보고 `[MASK]`에 들어갈 단어를 예측한다.

BERT는 문장 전체를 양방향으로 보기 때문에 이해 작업에 강하다.

### GPT

GPT는 문장을 생성하는 데 강한 모델이다.

주요 특징은 다음과 같다.

```text
Decoder-only Transformer
왼쪽에서 오른쪽으로 다음 토큰 예측
Next Token Prediction
생성, 대화, 코드 작성, 추론 작업에 강함
```

예를 들어 다음 문장이 있을 때:

```text
사용자가 악성 파일을
```

다음에 올 토큰을 예측한다.

```text
실행했다
다운로드했다
삭제했다
분석했다
```

GPT 계열 모델은 이 다음 토큰 예측을 매우 큰 규모로 학습하면서 강력한 생성 능력을 갖게 되었다.

## 9. Base Model과 Instruction Model의 차이

초기 GPT 같은 모델은 기본적으로 다음 토큰을 예측하는 모델이다.

이를 base model이라고 볼 수 있다.

Base model은 문장을 이어 쓰는 능력은 좋지만, 사용자의 지시를 항상 잘 따르지는 않는다.

예를 들어 사용자가 이렇게 물었다고 하자.

```text
다음 로그를 분석해줘.
```

Base model은 질문에 답하기보다, 인터넷 문서처럼 이어지는 텍스트를 생성할 수 있다.

그래서 등장한 것이 instruction tuning이다.

Instruction-tuned model은 질문-답변, 지시-응답 데이터로 추가 학습된 모델이다.

```text
사용자 지시를 이해하고
요구한 형식에 맞게 답변하고
대화형 assistant처럼 행동하도록 학습된다.
```

ChatGPT, Claude, Gemini 같은 서비스형 모델은 단순 base model이 아니라 instruction tuning, preference alignment, safety tuning 등이 적용된 모델이다.

## 10. ChatGPT 이후: LLM은 도구와 시스템으로 확장됨

ChatGPT 이후 LLM은 단순히 텍스트를 생성하는 모델에서, 외부 시스템과 연결되는 구조로 발전했다.

대표적인 흐름은 다음과 같다.

```text
RAG
Tool Calling
Agent
Function Calling
MCP
Multimodal
Reasoning Model
```

### RAG

RAG는 모델이 모든 지식을 내부 파라미터에 기억하도록 하는 대신, 외부 문서나 DB를 검색해서 답변에 활용하는 방식이다.

```text
사용자 질문
→ 관련 문서 검색
→ 검색 결과를 프롬프트에 포함
→ LLM이 답변 생성
```

네가 하는 보안 문서 RAG, TI DB 연계, pgvector 검색이 여기에 해당한다.

### Tool Calling / Agent

Tool Calling은 LLM이 필요할 때 외부 도구를 호출하는 방식이다.

예를 들어:

```text
Cortex XDR에서 최근 incident 조회
PostgreSQL에서 TI 정보 조회
웹 검색 수행
파일 읽기
SQL 실행
```

Agent는 이런 도구 호출과 판단 과정을 여러 단계로 수행하는 시스템이다.

네가 구성하는 `xdr-mcp-agent`, `t2s-agent`, `analysis-agent` 구조도 이 흐름에 있다.

## 11. 전체 흐름 다시 정리

LLM의 발전 흐름을 다시 보면 다음과 같다.

```text
1. 규칙 기반 NLP
   - 사람이 직접 규칙 작성
   - 문맥과 예외 처리에 약함

2. 통계 기반 NLP
   - 단어 등장 확률과 패턴 활용
   - 긴 문맥과 의미 이해에 약함

3. 단어 임베딩
   - 단어를 벡터로 표현
   - 의미적으로 가까운 단어를 가까운 벡터로 배치

4. RNN / LSTM
   - 문장을 순서대로 처리
   - 긴 문맥과 병렬 처리에 한계

5. Attention
   - 중요한 단어를 직접 참고
   - 긴 문맥 처리 성능 개선

6. Transformer
   - Attention 기반 구조
   - 병렬 학습과 대규모 모델 확장 가능

7. BERT / GPT
   - Transformer 기반 대형 언어 모델
   - BERT는 이해, GPT는 생성에 강함

8. Instruction LLM
   - 지시를 따르는 assistant 형태로 발전

9. RAG / Agent / Tool-use
   - 외부 지식과 도구를 활용하는 시스템으로 확장
```

## 12. 이번 편에서 꼭 기억할 것

이번 편에서 가장 중요한 내용은 아래다.

```text
LLM은 Transformer 기반의 대규모 언어 모델이다.

```
